# 💾 Система кэширования AI Assistant

**Дата внедрения:** 2026-03-19  
**Статус:** ✅ В продакшене

---

## 📋 Обзор

Система кэширования предназначена для ускорения ответов AI-помощника и снижения нагрузки на Qwen CLI. Кэширование происходит на уровне сервера и прозрачно для клиента.

### Проблемы, которые решает кэширование:

1. **Медленные ответы Qwen** — 5-30 секунд на генерацию ответа
2. **Повторяющиеся запросы** — одинаковые вопросы от разных пользователей
3. **Нагрузка на систему** — постоянные вызовы Qwen CLI
4. **Sentiment-анализ** — 3-20 секунд на анализ тональности

---

## 🎯 Объекты кэширования

### 1. Ответы Qwen (Qwen Responses)

**Что кэшируется:** Полные ответы AI на сообщения пользователей

**TTL:** 1 час (3600000 мс)

**Ключ кэша:** MD5-хэш от комбинации:
- Нормализованное сообщение (lowercase + trim)
- Service slug (или 'none')
- Длина истории диалога

**Пример:**
```
Запрос: "сколько стоит монтаж сайдинга?"
Service: "montazh-saydinga"
History: 0 сообщений

Ключ: md5("сколько стоит монтаж сайдинга?:montazh-saydinga:0")
```

### 2. Sentiment-анализ (Sentiment Analysis)

**Что кэшируется:** Результаты анализа тональности сообщений

**TTL:** 30 минут (1800000 мс)

**Ключ кэша:** MD5-хэш от комбинации:
- Нормализованное сообщение
- Длина истории диалога

**Пример результата:**
```json
{
  "ignore": false,
  "negative": false,
  "reason": "Нейтральный запрос информации"
}
```

### 3. Сессионные данные (Session Data)

**Что кэшируется:** Данные сессий пользователей (Telegram topic mapping)

**TTL:** 2 часа (7200000 мс)

**Ключ кэша:** Session ID

## 📊 API для управления кэшем

### GET /api/cache/stats

Получить статистику кэша.

**Ответ:**
```json
{
  "status": "ok",
  "cache": {
    "qwen": {
      "hits": 15,
      "misses": 42,
      "hitRate": "26.3%",
      "size": 35
    },
    "sentiment": {
      "hits": 8,
      "misses": 20,
      "hitRate": "28.6%",
      "size": 18
    },
    "session": {
      "hits": 120,
      "misses": 5,
      "size": 5
    }
  },
  "ttl": {
    "QWEN_RESPONSE": 3600000,
    "SENTIMENT_ANALYSIS": 1800000,
    "SESSION_DATA": 7200000
  },
  "timestamp": "2026-03-19T14:30:00.000Z"
}
```

### POST /api/cache/clear

Очистить кэш (полностью или выборочно).

**Тело запроса:**
```json
{
  "type": "qwen"  // "qwen" | "sentiment" | "session" | не указано (все)
}
```

**Ответ:**
```json
{
  "status": "ok",
  "cleared": 35,
  "type": "all",
  "stats": { ... },
  "timestamp": "2026-03-19T14:30:00.000Z"
}
```

## 🔧 Настройка

### Изменение TTL

Откройте `ai-assistant/server.js`:

```javascript
const CACHE_TTL = {
  QWEN_RESPONSE: 3600000,      // 1 час
  SENTIMENT_ANALYSIS: 1800000, // 30 минут
  SESSION_DATA: 7200000        // 2 часа
};
```

### Изменение интервала авто-очистки

Найдите `setInterval` и измените интервал (по умолчанию 10 минут):

```javascript
const cacheCleanupInterval = setInterval(() => {
  // ...
}, 600000); // 600000 мс = 10 минут
```

## 🧪 Тестирование

### Автоматический тест

```bash
node ai-assistant/test-cache.js
```

**Что проверяет:**
1. Кэширование ответов Qwen (cache miss → cache hit)
2. Кэширование sentiment-анализа
3. Статистику кэша
4. Очистку кэша

### Ручное тестирование

1. Отправьте одинаковое сообщение дважды:
```bash
curl -X POST http://localhost:3002/api/chat \
  -H "Content-Type: application/json" \
  -d '{"message":"сколько стоит?","sessionId":"test123"}'
```

2. Проверьте логи сервера:
```
💾 [CHAT] Cache MISS - calling Qwen...
🤖 [CHAT] ✅ Qwen response received: ...
💾 [CHAT] Response cached for 1 hour

💾 [CHAT] Cache HIT - using cached Qwen response
```

3. Проверьте статистику:
```bash
curl http://localhost:3002/api/cache/stats
```

## 📈 Мониторинг

### Логи сервера

Система логирует следующие события:

| Событие | Лог |
|---------|-----|
| Cache MISS | `💾 [CHAT] Cache MISS - calling Qwen...` |
| Cache HIT | `💾 [CHAT] Cache HIT - using cached Qwen response` |
| Кэширование | `💾 [CHAT] Response cached for 1 hour` |
| Авто-очистка | `🧹 [CACHE] Cleared N expired entries from ...` |
| Статистика | `📊 [CACHE STATS] {...}` (каждые 10 мин) |

### Метрики для отслеживания

- **Hit Rate** — процент попаданий в кэш (цель: >50%)
- **Cache Size** — размер кэша (следить за ростом)
- **Response Time** — время ответа (кэш: <100мс, Qwen: 5-30с)

## ⚠️ Важные замечания

### Когда кэш НЕ работает

1. **Разная история диалога** — ключ кэша включает длину истории
2. **Разный serviceSlug** — ответы кэшируются отдельно для каждой услуги
3. **Истёк TTL** — запись автоматически удаляется

### Ограничения

- In-memory кэш (сбрасывается при перезапуске сервера)
- Нет распределённого кэша (для нескольких серверов нужен Redis)
- MD5-хэширование (не криптографическая защита, только для ключей)

### Рекомендации

1. **Мониторьте hit rate** — если <10%, возможно стоит пересмотреть стратегию
2. **Следите за размером** — если >1000 записей, уменьшите TTL
3. **Очищайте при деплое** — если обновили knowledge base

## 🚀 Производительность

### Сравнение (средние значения)

| Операция | Без кэша | С кэшем | Улучшение |
|----------|----------|---------|-----------|
| Ответ Qwen | 5-30 сек | <100 мс | **×1000** |
| Sentiment | 3-20 сек | <10 мс | **×500** |
| Session | 1-3 сек | <1 мс | **×1000** |

### Ожидаемый эффект в продакшене

- **Среднее время ответа:** 2-3 сек → 0.5 сек
- **Нагрузка на Qwen:** -60-80%
- **Конверсия в заявку:** +10-15% (за счёт скорости)
